# 內容生產 · 批次 Dispatch 模板

> 用途：生成靈修內容批次時，套用此模板產出 Agent View dispatch prompt。
> 維護者：📝 內容生產視窗（模板本體）+ 💻 開發協調視窗（版控歸位）
> 最後更新：2026-05-30（批次 1/2a/2b/3 沉澱）

## 欄位填寫說明（兩類欄位，務必分清）
═══════════════════════════════════════════════════════════

### 🔧 機械欄位（未來可被進度 script 自動填）
這些欄位是「查表 + 算日期」就能得出，不需要動腦判斷：
- `{{書卷中文名}}`：例「帖撒羅尼迦前書」
- `{{書卷英文檔名}}`：hakkaac 用，例「1Thessalonians」
- `{{bible_com代碼}}`：BIBLE_LINKS 用，例「1TH」（注意不是 TH1）
- `{{章節清單}}`：例 TH1_1~TH1_5
- `{{章節key}}` / `{{章節號}}` / `{{排程日期}}`：每章一組
- `{{插入位置}}`：例「TH1 在 COL 後、TH2 前」
- `{{合併日標記}}`：哪些日期是合併日、SCHEDULE 要陣列化
- `{{commit_message}}`

### 🧠 判斷欄位（永遠留給人填 — 這是模板存在的意義）
這些欄位需要對經文、對玩家社群的理解，AI 不能自動產生：
- `{{THEME_HINT}}`：章節核心主題與張力
- `{{SCENARIO_FOCUS}}`：情境題聚焦哪段經文、往哪個切角發展
- `{{RELATIONAL_DIMENSION}}`：個人內在 / 與他人 / 與神（為主題多元性把關）
- `{{CAUTION}}`：⚠️ 雷區標註 — 哪些段落會傷害哪類玩家、必須怎麼避

> 🚩 紅線：判斷欄位永遠由 📝 內容生產視窗填寫，不可由 script 或
> subagent 自動產生。碰玩家內心的內容必須有人判斷。

延續紀律（批次 1/2a/2b/3 沉澱 — 固定，不要改）
═══════════════════════════════════════════════════════════

【紀律 1】commit 前必先確認分支
  git branch --show-current → 必須是 dev 才能繼續

【紀律 2】subagent 不可自加括號、不可切句、不可改標點
  經文必須整節或半節「連續取出」，不可拼接、不可省略。

【紀律 3】金句場景的結尾標點判斷
  - 原文頓號「、」或間隔號「．」（接續下節）→ 可改逗號「，」保留未完待續感
  - 原文句號「。」→ 維持句號
  - 原文冒號「：」→ 單獨呈現時可改句號斷句
  - subagent 自己判斷哪種結尾最適合該金句呈現

【紀律 4】SCHEDULE 全陣列格式
  - 單章日：['{{章節key}}']
  - 合併日：['{{章節key_A}}','{{章節key_B}}']

【紀律 5】BIBLE_LINKS 無空格格式
  '{{章節key}}':'https://www.bible.com/zh-TW/bible/46/{{bible_com代碼}}.{{章節號}}.CUV',

【紀律 6】全形空格「 神」一律移除（改為「神」無空格）

【紀律 7】寫 content.js 不走 sed 模式
  read 既有檔案 → 找到正確插入位置 → 用 Edit/Write 工具插入

【紀律 8】不要動 bible-game-v2.html（BOOKS 由開發協調統籌）

【紀律 9】不進版（不動 GAME_VERSION / VERSION_NOTES / changelog）
  GAME_VERSION 維持當前值 {{當前版本號}}

【紀律 10】commit 前再次 git branch 確認

════【本次任務】════
- 書卷：{{書卷中文名}}
- 章節：第 ▓章節號▓ 章
- 排程日期：▓排程日期▓
- 章節 key：▓章節key▓
- 章節主題提示：▓THEME_HINT▓          ← 🧠 判斷欄位
- 情境題主題建議：▓SCENARIO_FOCUS▓    ← 🧠 判斷欄位
- 關係向度：▓RELATIONAL_DIMENSION▓    ← 🧠 判斷欄位
- 雷區提醒：▓CAUTION▓                  ← 🧠 判斷欄位

════【必守紀律】════
1. 經文必須和合本逐字（含標點），用「，」「。」「；」新標點和合本風格
2. 引號內文字不能改寫、不能自加括號、不能切句拼接
3. baseItem.desc 與 bonusItem.desc 不能引用同一節經文
4. verse / baseItem.desc / bonusItem.desc 三處引用必須都在本章範圍內
5. choices 涵蓋不同信仰成熟度與誠實程度
6. 至少一個選項給「老實說我做不到」的人（誠實面對軟弱）
7. 每個 response 結尾留下「情緒有重量的小步」——
   一句真實會說出口的話，或一個有方向感的問題，不是「想一想」空話
8. responses 溫暖、有洞見、像朋友同行、不說教
9. guide.outline 用 ✦ 標記情境題聚焦段落
10. guide.focus 明確指出情境題聚焦的經文範圍
11. 全形空格「 神」移除（用「神」無空格）
12. 金句結尾標點按「金句場景」判斷

════【格式範本】════

{
  chapter: '▓章節key▓', sceneEmoji: '...',

  guide: {

  },
  verse: '「...」',
  verseRef: '—— {{書卷中文名}} ▓章節號▓:N',
  scene: '...',
  q: '...',
  choices: [
    { k: 'A', text: '...' },
    { k: 'B', text: '...' },
    { k: 'C', text: '...' },
    { k: 'D', text: '...' }
  ],
  responses: { A: '...', B: '...', C: '...', D: '...' },
  reflectionTitle: '...',
  reflection: '...\n\n...',
  baseItem: { emoji: '...', name: '...', desc: '「...」', slot: '...' },
  bonusItem: { emoji: '...', name: '...', desc: '「...」', slot: '...' }
}

【裝備 slot 分類】
- hat：象徵性標誌、身份、祝福
- body：角色外袍、囚衣、身分象徵
- hand：實際拿在手上的物件
- bg：場景意象、地圖、海浪、火焰

════【執行流程】════

Step 1【生成專員】
  - 用 WebFetch 抓和合本經文
    主來源：https://hakkaac.org/Bible/Bible/CUT/CUT_{{書卷英文檔名}}.html
    備援 1：https://wd.bible/verse/{{bible_com代碼小寫}}.▓章節號▓.1.cunps
    備援 2：https://cnbible.com/{{書卷英文小寫}}/▓章節號▓.htm
  - hakkaac 是「神版傳統」用頓號「、」與間隔號「．」，
    轉換為新標點和合本「，」「。」「；」風格
  - 讀整章，找出 verse / baseItem.desc / bonusItem.desc 三節（不重複、不跨章）
  - 注意 ▓THEME_HINT▓ / ▓SCENARIO_FOCUS▓ / ▓RELATIONAL_DIMENSION▓ / ▓CAUTION▓

Step 2【審查員】跑完整 A-F 審查清單：

  ─ A. 格式完整性 ─
  □ 所有必要欄位齊全 □ choices 剛好 4 個 □ responses 對應 4 個
  □ guide 含 intro/outline/focus □ baseItem 與 bonusItem 含 emoji/name/desc/slot

  ─ B. 和合本核對 ─
  □ verse / baseItem.desc / bonusItem.desc 皆真實和合本原文（含標點）
  □ 三處引用都在本章範圍內 □ baseItem 與 bonusItem 不引用同一節
  □ 全形空格「 神」已移除 □ 沒有自加括號、沒有切句拼接
  □ 金句結尾標點符合「金句場景」判斷

  ─ C. 情境題品質 ─
  □ q 連結玩家真實生命，非純知識測驗
  □ 四選項涵蓋不同信仰成熟度與誠實程度
  □ 至少一個「老實說做不到」選項
  □ 每個 response 結尾有「情緒有重量的小步」
  □ responses 無說教語氣
  □ ▓CAUTION▓ 列出的雷區皆已避過

  ─ D. 裝備設計 ─
  □ baseItem 名稱與 desc 同神學層次 □ bonusItem 同
  □ 三處引用不跨章 □ slot 分類合理

  ─ E. 人話自檢 ─
  □ choices 是真實的人會說出口的話
  □ responses 自然，不像官方聲明或勵志貼文
  □ scene 讓不熟悉聖經的人也能理解情境

  ─ F. guide 導讀品質 ─
  □ intro 幫助玩家對整章有初步認識
  □ outline 清楚列出結構重點，含 ✦ 標記 □ focus 說明情境題聚焦點

Step 3【迴圈】任何項 ❌ → 切回生成專員只修問題處；全 ✅ → Step 4

Step 4【輸出】
  ✅ 審查通過 · 共迭代 [N] 輪
  章節 key：▓章節key▓ / 排程日期：▓排程日期▓
  [完整 JS 物件，從 { 到 }]
  審查摘要：迭代輪數 / 修改項目 / 和合本核對 / 三處經文出處 /
            關係向度 / 雷區避過項目

════【重要：不要 commit、不要寫檔】════
只負責生成與審查，不動任何檔案、不跑 git。
通過的章節物件以純文字輸出給 parent agent 收齊。
