## 1. 數據來源

### 1.1 自動化（指令）

| 指令 | 來源 | 涵蓋 |
|---|---|---|
| `npm run analyze` | Firestore | feedback、users、章節品質①-④、裝備⑤、默想歷史⑥ |
| `npm run logs [days]` | Cloud Function logs | aiReflection 呼叫量、成功率、錯誤明細（預設 1 天）|
| `npm run line-logs [days]` | Cloud Function logs | lineLogin 成功率、失敗時段（預設 1 天）|

### 1.2 手動（GA4 匯出）

需要去 [analytics.google.com](https://analytics.google.com) 屬性 `G-HZ3EGYB8BB` 匯出：
- **報表數據匯報.xlsx**：總覽（活躍、新使用者、參與時間、來源、國家、頁面、事件、留存）
- **Firebase_總覽.csv**：DAU/WAU/MAU 趨勢、cohort 留存
- **「瞭解網站和/或應用程式流量」總覽報表.csv**：城市、語言、DAU/MAU 比例
- **「查看使用者參與度和留存率」總覽報表.csv**：cohort 第 1 天 / 第 7 天留存
- **技術詳情_瀏覽器.csv**：瀏覽器分布
- **事件_事件名稱.csv**：每事件的次數 + 觸發人數

放在專案根目錄，下次更新本文件時讀取分析。

## 7. 重要洞察記事

### 2026-05-01：首次完整解讀

#### 洞察 1：「週日教會分享當下進來，但隔天忘記」
- 4/26 週日 16 人新加入，Day 1 只剩 1 人（6%）
- **平日進來的反而留得住**（4/24 週五 91%）
- → 這是核心流失點，不是內容問題，是「習慣養成問題」
- → 待解：Cloud Messaging 推播 / LINE 官方帳號每日提醒

#### 洞察 2：21 位重度分享者是成長引擎
- 21 個玩家分享 77 次（每人 3.67 次）
- 這群「忠實傳教者」是 4/26 週日 16 新人的來源
- → 待做：從 `npm run analyze` 找出 shareCount 排行，識別這 21 人

#### 洞察 3：63% 玩家在 LINE app 內開
- 49 / 78 人從 LINE 訊息點進來
- 不會加書籤、收不到 Web Push
- → 推播策略應走 LINE 官方帳號（CLAUDE.md 待開發項目）

#### 洞察 4：完成靈修人數比漏斗高 54%
- 漏斗算 26 人 vs 事件人數實際 40 人
- 因為漏斗強制 read_chapter 在前，但這是 OPTIONAL 步驟
- → 重新解讀過去的判讀

#### 洞察 5：玩家偏好「短而真誠」的默想
- 默想字數中位 25 字、p90 74 字、無人寫超過 200 字
- 0% 改寫率：玩家送了就不回頭
- → 設計意圖驗證：UI 不需強迫長文

#### 洞察 6：曠野呼聲入口隱藏
- 全月 2 筆回饋
- → 待決議：要 surface 入口（風險：太頻繁打擾）還是維持現狀（理由：沒人抱怨 = 沒大問題）

## 9. 怎麼更新這份文件

### 每週快速更新
1. 跑 `npm run analyze` 抓 Firestore snapshot
2. 跑 `npm run logs 7` 抓過去 7 天 AI 服務狀態
3. 對照本文件「§6 內容與服務品質」更新數字
4. 「§7 重要洞察記事」加新一筆 `### YYYY-MM-DD：……`

### 每月深度更新（含 GA4）
1. GA4 → 報表 → 各區塊 → 右上「分享 → 下載檔案」匯出 CSV / xlsx
2. 把檔案放專案根目錄
3. 跟 Claude Code 說「請更新 data-insights.md」
4. 重點更新區塊：§2 規模、§3 玩家屬性、§4 事件、§5 留存

### 重大事件即時記錄
- 部署、bug 修復、產品決策 → 直接寫進「§7 洞察記事」

### 修改原則
- **不刪舊資料**：洞察是 timeline，舊的有歷史價值
- **健康度紅綠燈會被覆蓋**：以最新快照為準
- **每次更新最後改「最後更新」日期**
