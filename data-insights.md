## 7. 重要洞察記事

### 2026-05-01：首次完整解讀

#### 洞察 1：「週日教會分享當下進來，但隔天忘記」
- 4/26 週日 16 人新加入，Day 1 只剩 1 人（6%）
- **平日進來的反而留得住**（4/24 週五 91%）
- → 這是核心流失點，不是內容問題，是「習慣養成問題」
- → 待解：Cloud Messaging 推播 / LINE 官方帳號每日提醒

#### 洞察 2：21 位重度分享者是成長引擎
- 21 個玩家分享 77 次（每人 3.67 次）
- 這群「忠實傳教者」是 4/26 週日 16 新人的來源
- → 待做：從 `npm run analyze` 找出 shareCount 排行，識別這 21 人

#### 洞察 3：63% 玩家在 LINE app 內開
- 49 / 78 人從 LINE 訊息點進來
- 不會加書籤、收不到 Web Push
- → 推播策略應走 LINE 官方帳號（CLAUDE.md 待開發項目）

#### 洞察 4：完成靈修人數比漏斗高 54%
- 漏斗算 26 人 vs 事件人數實際 40 人
- 因為漏斗強制 read_chapter 在前，但這是 OPTIONAL 步驟
- → 重新解讀過去的判讀

#### 洞察 5：玩家偏好「短而真誠」的默想
- 默想字數中位 25 字、p90 74 字、無人寫超過 200 字
- 0% 改寫率：玩家送了就不回頭
- → 設計意圖驗證：UI 不需強迫長文

#### 洞察 6：曠野呼聲入口隱藏
- 全月 2 筆回饋
- → 待決議：要 surface 入口（風險：太頻繁打擾）還是維持現狀（理由：沒人抱怨 = 沒大問題）

### 2026-05-03：第二波擴張潮 + 服務品質驗證

#### 洞察 7：5/3 +13 新玩家來自教會領袖管道（**非目標客群，預期流失**）
- **直接觸發事件**：5/3 教會長執會，牧師於會中分享此遊戲；會後 James 把連結分享至「長老執事同工」LINE 群組
- 13 人來自**教會領袖層信任管道**：長執同工 → 各自小組
- ⚠️ **但這群人不是產品目標客群**：長老執事大多已有穩定靈修習慣，這個遊戲的初衷是「**幫助沒有穩定靈修習慣的朋友**」，**不是改變既有靈修者的習慣**
- 因此預期：這 13 人 Day 1 流失率會高（不是 bug，是合理結果）；他們的價值在「成為傳教者」（再分享給核心目標族群），不在「自己留下來玩」
- → 戰略意義：未來重要里程碑可借長執會 / 主日 / 同工群組擴散，但**留存指標應排除這類「展示型用戶」**單獨計算
- → 真正要追蹤的：**他們再分享後，那些「之前不太靈修」的朋友有沒有留下來**（這群在數據上難辨識，可能要從 streak ≥ 3 + 沒有 Google 帳號的 LINE 用戶來篩）

#### 洞察 8：默想字數翻倍成長，打臉「短而真誠」假設
- 5/1 中位 25 / p90 74 / 最長 105 → 5/3 中位 29 / p90 **106** / 最長 **232**（翻倍）
- 累積筆數 26 → 51 也翻倍
- 5/1 的設計意圖驗證為「玩家偏好短文，不需強迫長文」— 但實際玩家自發寫越來越長
- → **不要做相反方向**（限制字數）；可考慮 UI 提示「想寫多長都可以」鼓勵深入
- → 也可能與「教會新人加入後，老玩家深度增加」有關（待交叉看老玩家字數變化）

#### 洞察 9：2 次 retry + jitter 部署實證有效
- 5/3 過去 24h aiReflection：**30 次呼叫，100% 真實 AI 回應**
- LINE 登入同期 30 次，100% 成功
- 但部署後累積（4/29 ~ 5/3 共 26 筆）仍 88%（fallback 3）— 中間幾天偶有失敗
- → 「2 次 retry + jitter」相對「1 次 retry」確實救回更多，但非 100% 完美

#### 洞察 10：ROM10 章節 fallback 異常集中
- 全歷史 12 次 fallback 中，**ROM10 占 7 次（58%）**
- 部署後 4/29+ 仍見 ROM14（2 次）、ROM15（1 次）— 都集中在 Romans 中段
- ROM10 主題：因信稱義（救恩 / 信心 / 律法）
- 假設：玩家默想中含信仰判斷詞彙 → Gemini safety filter 觸發 → 拿到 fallback 文字
- → 待做：檢查 ROM10 在 content.js 的 system prompt；可考慮為這類章節準備專屬 fallback 訊息

## 9. 怎麼更新這份文件

### 每週快速更新
1. 跑 `npm run analyze` 抓 Firestore snapshot
2. 跑 `npm run logs 7` 抓過去 7 天 AI 服務狀態
3. 對照本文件「§6 內容與服務品質」更新數字
4. 「§7 重要洞察記事」加新一筆 `### YYYY-MM-DD：……`

1. GA4 → 報表 → 各區塊 → 右上「分享 → 下載檔案」匯出 CSV / xlsx
2. 把檔案放專案根目錄
3. 跟 Claude Code 說「請更新 data-insights.md」
4. 重點更新區塊：§2 規模、§3 玩家屬性、§4 事件、§5 留存

### 重大事件即時記錄
- 部署、bug 修復、產品決策 → 直接寫進「§7 洞察記事」

### 修改原則
- **不刪舊資料**：洞察是 timeline，舊的有歷史價值
- **健康度紅綠燈會被覆蓋**：以最新快照為準
- **每次更新最後改「最後更新」日期**
