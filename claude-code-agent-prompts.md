## 🔄 主提示詞（單一 session 雙角色迴圈版）

**使用方式**：打開 Claude Code，把以下提示詞完整貼入，填好章節資訊後送出。

```
請閱讀專案根目錄的 CLAUDE.md，了解「靈修冒險」遊戲的完整背景與設計規範。

你這次要同時扮演兩個角色，在同一個 session 裡輪流切換，直到內容通過審查才停止：

───────────────────────────────
角色一：【生成專員】
負責根據設計規範生成靈修內容
───────────────────────────────
角色二：【審查員】
負責逐項檢查內容品質，發現問題就回傳給生成專員修改
───────────────────────────────

【本次任務】
書卷：[填入，例如：哥林多前書]
章節：[填入，例如：第5章]
排程日期：[填入，例如：2026-05-12]
章節 key：[填入，例如：COR1_5]

【章節 key 命名規則】
・使徒行傳 → 數字（例如：10, 11, 27）
・羅馬書 → ROM + 章數（例如：ROM1, ROM16）
・哥林多前書 → COR1_ + 章數（例如：COR1_1, COR1_10）
・之後的書卷依此類推（COR2_, GAL, EPH...）

════════════════════════════════
第一輪：【生成專員】執行
════════════════════════════════

請以【生成專員】身份，按照以下步驟生成內容：

步驟 1｜確認和合本原文
先確認本章節的和合本內容，特別記下適合引用的金句。
可透過 Bible.com 查詢原文（遊戲 BIBLE_LINKS 裡有每章連結）。

步驟 2｜撰寫 guide 導讀
- readTime：預估閱讀完整章節的分鐘數（3-6）
- intro：2-3句，整章大意，讓不熟悉聖經的人也能理解
- outline：3-4個段落重點，格式為「節數｜重點」
- focus：1句，說明今日情境題聚焦在哪裡

步驟 3｜撰寫核心內容
- verse：今日金句，必須是和合本原文，一字不差
- verseRef：出處（例如：—— 哥林多前書 5:7）
- scene：2-4句，描述今日情境背景

步驟 4｜撰寫情境題
- q：開放性問題，連結玩家真實生命
- choices：4個選項（A/B/C/D），涵蓋不同信仰成熟度
  ・至少一個選項給「老實說我做不到」的人
  ・用真實的人話，不用宗教術語
- responses：對應4個選項的回應
  ・溫暖有洞見，像朋友同行，不說教
  ・每個回應最後留下「一個情緒有重量的小步」
   （真實的人會說出口的話，或有方向感的問題）

步驟 5｜撰寫默想
- reflectionTitle：2-4字主題
- reflection：兩段，第一段聖經背景，第二段個人問題

步驟 6｜設計裝備
- baseItem：完成靈修獲得，emoji + name + desc + slot
- bonusItem：填寫默想後獲得（稀有），emoji + name + desc + slot
- desc 必須是真實的和合本原文，一字不差
- 名稱和 desc 必須在同一個神學層次
- 判斷本章是否適合做性別差異裝備：
  ・若經文有明確的性別角色意象（如才德婦人、勇士）→ 可用性別格式
  ・性別格式：{ default:{...}, m:{...}, f:{...} }
  ・大部分章節不需要，只在自然適合時才用

完成後，輸出完整的 JavaScript 物件。

════════════════════════════════
第二輪：【審查員】執行
════════════════════════════════

【生成專員】完成後，立刻切換成【審查員】身份，逐項審查：

─── A. 格式完整性 ───
□ 所有必要欄位是否齊全
□ choices 是否剛好 4 個（A/B/C/D）
□ responses 是否對應 4 個選項
□ guide 是否有 intro / outline / focus
□ baseItem 和 bonusItem 是否有 emoji / name / desc / slot

─── B. 和合本核對（上網驗證）───
□ 透過 Bible.com 或 BIBLE_LINKS 查詢原文，逐字比對以下欄位：
□ verse 是否為真實和合本原文（用字、標點是否正確）
□ baseItem.desc 是否為真實和合本原文
□ bonusItem.desc 是否為真實和合本原文
□ 截斷經文等同引用錯誤，整節太長要換節而非截半
□ 兩個裝備不能拆同一節經文

─── C. 情境題品質 ───
□ q 是否連結玩家真實生命，而非純知識測驗
□ 四個選項是否涵蓋不同信仰成熟度
□ 是否有「老實說我做不到」的選項
□ 每個 response 是否有「情緒有重量的小步」
□ responses 是否有說教語氣（有 → 需修改）

─── D. 裝備設計 ───
□ baseItem 名稱和 desc 是否同神學層次
□ bonusItem 名稱和 desc 是否同神學層次
□ baseItem.desc 引用的經文是否在本章範圍內（不可跨章引用）
□ bonusItem.desc 引用的經文是否在本章範圍內（不可跨章引用）
□ verse 引用的經文是否在本章範圍內（不可跨章引用）

─── E. 人話自檢 ───
□ choices 的文字是真實的人會說出口的嗎
□ responses 的文字自然，不像官方聲明或勵志貼文嗎
□ scene 讓不熟悉聖經的人也能理解情境嗎

─── F. guide 導讀品質 ───
□ intro 是否幫助玩家對整章有初步認識
□ outline 是否清楚列出本章結構重點
□ focus 是否說明了今天情境題的聚焦點

════════════════════════════════
迴圈規則
════════════════════════════════

審查完成後：

如果有任何 ❌：
→ 切換回【生成專員】
→ 只修改有問題的部分，不動通過的部分
→ 修改完畢，切換回【審查員】重新審查有問題的項目
→ 重複以上步驟，直到所有項目通過

如果全部 ✅：
→ 輸出以下內容給我確認：

---
✅ 審查通過 · 共迭代 [N] 輪

[完整的 JavaScript 物件]

審查摘要：
・迭代輪數：N 輪
・修改項目：[列出曾修改過的項目]
・和合本核對：已確認 verse / baseItem.desc / bonusItem.desc
---

等我確認後，再更新 content.js 並 git push。
```

## 📋 使用流程

```
Step 1｜打開 Claude Code
Step 2｜貼上主提示詞，填入書卷/章節/日期/key
Step 3｜送出，等待 agent 自動迭代
Step 4｜收到「✅ 審查通過」後過目內容
Step 5｜確認沒問題，回覆「可以，請更新」
Step 6｜Claude Code 更新 content.js 並 git push
```

## 🗒️ 補充說明

**關於迭代輪數**
一般內容 1-2 輪就會通過。如果超過 3 輪還有問題，通常是和合本經文確認困難，可以手動指定經文讓它繼續。

**關於 Claude Code Agent Teams（實驗功能）**
Claude Code v2.1.32+ 有內建 Agent Teams，可以讓兩個真正獨立的 agent 互相通訊。
目前仍是實驗功能、有已知限制，待穩定後可升級為真雙 agent 架構。
啟用方式：在 settings.json 加入 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

**關於這份提示詞的存放**
建議把這份檔案上傳到 GitHub repo 跟 CLAUDE.md 放在同一層，
方便未來共同開發者直接使用。
